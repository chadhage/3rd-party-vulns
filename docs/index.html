What this repo is — and isn't

Read this first. The tool reports two different kinds of data, and only one of them is “vulnerabilities.”

✅ What this repo is

  • A read-only Azure Monitor workbook that reports two data sets behind one domain selector.
  • Software domain: genuine third-party CVE findings from Microsoft Defender Vulnerability Management, via Azure Resource Graph, scoped to non-Microsoft vendors.
  • Entra domain: an inventory & posture view of app registrations, enterprise apps, credentials, delegated consents, and app-only Microsoft Graph permissions.
  • Deployable as code (Bicep), with a matching Postman collection and a 75-test suite (including Graph transform contract tests).
  • Least-privilege and low blast radius: delegated read in the workbook; no data is written or exported.

⛔ What this repo isn't

  • Not a vulnerability scanner for Entra/Graph — Microsoft Graph does not expose app CVEs; the Entra tabs are inventory and configuration, not findings.
  • Not a vulnerability-management product: no risk scoring, prioritization, remediation, ticketing, alerting, or tracking.
  • Not a complete permissions audit — the app-permissions tab covers Microsoft Graph only, not SharePoint, Exchange, or Key Vault.
  • Not turnkey for every tenant: recommendations need Entra ID P1/P2 (beta endpoint) and CVEs need Defender for Servers P2 / MDVM onboarding.
  • Not a credential-expiry monitor — secret/certificate end-dates are listed, but not flagged as expired or expiring.

Deploy the workbook

Use Bicep, or import the JSON directly in the Azure portal.

Option A — Bicep

# Create a resource group, then deploy
az group create --name rg-security-workbooks --location westeurope

az deployment group create `
  --resource-group rg-security-workbooks `
  --template-file workbook/deploy.bicep

Outputs workbookResourceId and workbookName.

Option B — Portal import

Open the Advanced Editor

Azure Portal → Monitor → Workbooks → + New → Advanced Editor (</>).

Paste the JSON

Paste the contents of workbook/third-party-vulnerabilities.workbook.json.

Save

Apply → Done Editing → Save.

Prerequisites: Azure CLI signed in (az login). Entra domain needs Microsoft Graph delegated read (Global Reader or Security Reader). The Entra recommendations tab requires Entra ID P1/P2. The Software domain needs Defender for Servers Plan 2 / MDVM and Reader on the selected subscriptions. Saving needs Monitoring Contributor or Workbook Contributor.

Entra ID Applications

Each tab runs a Microsoft Graph query and renders the result as a table.

TabGraph endpoint
Entra recommendationsGET /beta/directory/recommendations
Enterprise application inventoryGET /v1.0/servicePrincipals
Application credentialsGET /v1.0/applications
Delegated permission grantsGET /v1.0/oauth2PermissionGrants
Microsoft Graph app permissionsGET /v1.0/servicePrincipals(appId='00000003-…')/appRoleAssignedTo
High-privilege Microsoft Graph app roles are highlighted: Directory.ReadWrite.All, RoleManagement.ReadWrite.Directory, Application.ReadWrite.All, AppRoleAssignment.ReadWrite.All  ·  Mail.ReadWrite, Files.ReadWrite.All. Only app-only permissions against Microsoft Graph are listed; other APIs (SharePoint, Exchange, Key Vault) are not shown.

Software Vulnerabilities (Defender)

Set the Subscriptions, Severity, and Vendor filters, then select a tab.

TabContent
CVE FindingsEach CVE finding with severity, CVSS, software, vendor, patchability, and resource.
Vulnerable SoftwareSoftware inventory aggregated by vendor/product with device counts.
Affected DevicesDevices with Critical/High counts and distinct CVE counts.
End-of-Support SoftwareProducts reported as unsupported or approaching end of support.

Data sources (Azure Resource Graph): microsoft.security/assessments/subassessments for CVE findings and microsoft.security/softwareinventories for inventory and end-of-support. Queries use coalesce(…) for property names that vary by plan and filter out vendors containing microsoft.

Microsoft Graph via Postman

Call the same five Graph endpoints from a client using app-only (client-credentials) auth.

Setup

Register an app

Add a client secret and the Graph application permissions DirectoryRecommendations.Read.All, Application.Read.All, Directory.Read.All. Grant admin consent.

Import both files

Import the collection and environment from postman/.

Set variables

Set tenantId, clientId, and clientSecret in the environment.

Send a request

The pre-request script fetches and caches an app-only bearer token automatically.

Permission models

Workbook (delegated): runs as the signed-in user. A directory role such as Global Reader or Security Reader is enough — no app registration or secret.

Postman (application): runs as an app registration with app-only permissions and a client secret, not tied to a user, requiring admin consent.

Pagination: responses return a value array; when more results exist an @odata.nextLink is stored in nextLink. Send the Next page request until it is no longer set.

Tests & coverage

Pester validates the artifacts; a Node harness executes the embedded Postman JavaScript with enforced coverage.

Run the suite

# From the repository root
Invoke-Pester -Path ./tests -Output Detailed

Requires PowerShell 7+, Pester 5+, Node.js, and Azure CLI (to compile the Bicep template).

What is covered

JSON validity (strict) Workbook structure Content quality Data robustness invariants Graph transform contract Bicep compile & settings Postman collection README accuracy Behavioral coverage ≥80%

The behavioral harness runs the embedded Postman scripts against mocks at 100% line, ~90% branch, and 100% function coverage, enforced at a ≥80% threshold.
